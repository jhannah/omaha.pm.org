Well, heck then why not do all of your configuration in an XML file? Then you could just have an set of elements slerped in.

<elements>
 <element>the good</element>
 <element>the bad</element>
 <element>the ugly</element>
</elements>

Loading it up varies by the parser used, of course. With Windows, I might choose to use MSXML. I might also choose Perl::XML. That has it's own support mailing list too.

Beauty is that this is simple enough for even windows user to understand while being completely Inter-op friendly, self-documenting yet ozzing all the booty-smacking jigginess that greekdom can muster.

On Wed, 18 Jun 2003 thehaas@xxxxxxxxxx wrote:
> > Why is @array an array? Why isn't it a hash?
>
> Because the array is going to be a configurable parameter in my script --
> I think it would be easier for my customer to change this:
> @array = ("fred","barney", . . .);
>
> Then:
> %hash = ("fred"=>1,"barney"=>1, . . . );
>
> > if ($hash{$bad}) {
> > # Do something
> > }
>
> But that is better -- too bad I can't use it.

Sure you can, Mike! -grin-

Idea #1: Do you really want your customers configuring the script? Maybe
they should just configure a config.txt file or something:

config.txt:
-----------
fred
barney
wilma
betty
bambam
-----------

Your script:

open (IN, "config.txt");
while (<IN>) {
   chomp;
   $hash{$_} = 1;
}
close IN;

if ($hash{$bad}) {
   # Do something
}

...

Idea #2: Your code could look like this:

# --------------------------------------------
# Dear customer: Please place your list here:
# --------------------------------------------
my @array = ("fred", "barney", "wilma", "betty", "bambam");
# --------------------------------------------
# Thank you! -Mike 6/18/2003
# --------------------------------------------

my %hash = map { $_ => 1 } @array;

if ($hash{$bad}) {
   # Do something
}

...

Idea #3: Just like idea 2, but if the list has no spaces this'll look
cleaner for your customer:

my @array = qw( fred barney wilma betty bambam );

...
