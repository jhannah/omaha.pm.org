"Matt Arnold" <marnold@xxxxxxxxx> wrote:
> Good question.  But please don't learn to code by looking at CGI.pm.  :-)
> 
> In an effort to make the monolithic CGI.pm module load as quickly as
> possible, many of the functions are initially read in as strings.
> (Apparantly Perl can do this faster than parsing/compiling the code.)  Then
> some hacks and black magic are applied to compile the various functions
> on-demand at the time they're needed.  It's all done for the sake of speed.
> And if you use all the functions in CGI.pm anyway, you're effectively just
> delaying compilation until later since all the code will eventually get
> compiled anyway.  But if you only use a few functions in CGI.pm, then
you'll
> come to appreciate the time savings.
> 
> But the best way to make your web pages fast -- particularly for
> apps/modules consisting of hundreds of thousands of lines of Perl -- is to
> use mod_perl.  This Apache module allows your Perl code to be compiled once
> at server startup and persist.  Although, it can be a bit of a hassle to
> build/compile a mod_perl-enabled Apache -- particularly if you didn't build
> your own Perl installation.  See perl.apache.org for more info.
>
