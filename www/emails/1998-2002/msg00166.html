> All mod_perl installations are horror stories.  :-)

> Well, lets just say it took me several iterations of:
> 
> 1) get jazzed about using mod_perl
> 2) try to install and set up mod_perl
> 3) watch my system spew errors and go up in flames
> 4) get mad and vow to never go near mod_perl again
> 5) wait a few months
> 6) goto step 1

Remarkable ... exactly my experience as well


> After doing this for a while, I finally got mod_perl to work.  It's not too
> bad if you know a few simple tricks.  And now that I know these tricks, I
> can get mod_perl up and running in a jiffy -- running perfectly and
> reliably.

I've gone through enough iterations of steps 1-6 that I can get mod_perl
up and running perfectly ... but I'm not at all yet comfortable with it
... I'm getting nervous right now, just thinking about it :)  But hey
... at least I don't have to rush to the bathroom anymore whenever I
hear someone mention it!


> 2) Build your own apache with mod_perl statically linked in.  Don't trust
> the RPM's.  And don't try to use apache's DSO support for a mod_perl.so
> module.  Instead, simply download the source for apache and the source for
> mod_perl, untar them both, then use mod_perl's configure/build/install
> process to set up and install your mod_perl-enabled apache.  This approach
> works most reliably on all *NIX's.

I would definitely concur on the RPM's ... even when you're working with
an actual Red Hat Linux distro.

I haven't had problems going the DSO route.  Do you advise against this
strictly on issues of performance?  I'm usually kicking around on a
'non-production' box, and generally haven't had to worry about DSO
performance penalties ... but I'm sure they're there :(
